The PCA exam tests your ability to apply **Google-recommended best practices**. Here is how you should modify the lab to align with those standards.

## 1. Security: Principle of Least Privilege
In the current lab, you are using `gcloud auth application-default login`, which often grants broad permissions tied to your user account.
* **PCA Enhancement:** Create a dedicated **Service Account** for Terraform.
* **Action:** * Create a service account with only the roles needed (e.g., `roles/storage.admin` for the state bucket and `roles/compute.admin` for the infrastructure).
    * In your `0-authentication.tf`, reference a Service Account key or, better yet, run Terraform from a Bastion host/Cloud Shell using the service account's identity to avoid local keys.

## 2. Reliability: State File Versioning and Locking
A critical PCA concept is **State Integrity**. If two engineers run Terraform at the same time, the state file can become corrupt.
* **PCA Enhancement:** Enable **Object Versioning** on the GCS bucket.
* **Action:**
    * Modify the bucket creation step (or use Terraform to manage the bucket itself) to include `versioning { enabled = true }`. This allows you to recover if a `terraform apply` goes wrong.
    * Note: GCS natively handles state locking, but ensuring versioning is enabled is the "Architect" way to protect data.

## 3. High Availability (HA) & Disaster Recovery
The current lab uses a single-region bucket and hardcodes a zone (`us-east4-a`) for a disk. 
* **PCA Enhancement:** Design for regional or multi-regional resilience.
* **Action:**
    * **Bucket:** Use a **Multi-Regional** or **Dual-Region** bucket for the Terraform state to ensure the state is available even if a whole region goes down.
    * **Infrastructure:** Instead of a standalone `google_compute_disk`, use a **Regional Persistent Disk** (replicated across two zones) to ensure data availability during a zonal outage.

## 4. Infrastructure as Code (IaC) Best Practices
The lab currently has you creating the bucket manually in the GUI. 
* **PCA Enhancement:** **Bootstrapping**. An architect tries to automate everything.
* **Action:** Use a "Bootstrap" script or a separate Terraform configuration to create the GCS bucket and the VPC. This ensures the entire environment is reproducible.

```hcl
terraform {
  backend "gcs" {
    bucket  = "prod-tf-state-rahde-jones"
    prefix  = "terraform/state/vstream-app" # Organized by application
  }
}
```

## 5. Transitioning from AWS to GCP (Your "Note to Self")
Since you are converting AWS labs, remember the PCA exam expects you to know the GCP equivalents for HA:
* **AWS VPC Peering** $\rightarrow$ **VPC Network Peering**.
* **AWS EBS (Direct)** $\rightarrow$ **GCP Persistent Disk**.
* **AWS S3** $\rightarrow$ **Cloud Storage** (remember that GCS is global, S3 is regional).

By adding **IAM restrictions**, **State Versioning**, and **Variables**, you transform a simple "homework assignment" into a "Production-Ready Deployment" which is exactly what the PCA exam looks for.
